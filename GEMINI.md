# Project AI Context: PRM (D&D Session Manager)

## 1. Загальний опис
**PRM** — це локальний веб-застосунок для майстрів настільної гри Dungeons & Dragons (D&D 5e). Він призначений для планування кампаній, керування сесіями, ведення бойових зіткнень та автоматизації генерації контенту за допомогою ШІ.

## 2. Технологічний стек
- **Frontend:** React 18, Vite, SCSS (BEM-методологія).
- **Backend:** Node.js, Express (локальний сервер).
- **База даних:** Відсутня (використовується файлова система JSON у папці `data/`).
- **AI:** Google Gemini API (модель `gemini-2.5-flash`).
- **PWA:** Service Worker для офлайн-доступу до локальної бази даних.
- **Ключові бібліотеки:** `react-markdown` (рендеринг тексту), `react-list` (віртуалізація списків).

## 3. Архітектура проєкту
### Клієнтська частина (`src/`)
- **Components:** Модульні компоненти (Panel, Button, Input, Modal, DiceCalculator).
- **Features:**
  - **Bestiary:** Локальна база даних монстрів, серверний пошук, віртуалізація.
  - **Spells:** Локальна база даних заклинань, віртуалізація.
  - **DiceCalculator:** Парсинг формул (regex), історія кидків, гарячі клавіші (`Ctrl+D`).
  - **AiAssistantPanel:** Інтеграція з Gemini для генерації сюжету та промптів для зображень.
- **Utils:** `diceParser.jsx` — центральний вузол для обробки тексту, регулярні вирази для кубиків, бонусів та посилань на заклинання (`{@spell Name}`).

### Серверна частина (`server/`)
- **server.js:** REST API для роботи з локальними файлами (CRUD для кампаній та сесій).
- **aiService.js:** Прошарок для спілкування з Gemini AI. Налаштовані системні інструкції для української мови та специфічних JSON-структур.

## 4. Специфічна логіка
### Стратегія кешування (Service Worker)
- **Стратегія:** Cache First, falling back to network.
- **TTL:** 1 тиждень (через кастомний заголовок `sw-fetched-on`).
- **Granular Caching:** При отриманні списків (spells), SW розбиває масив і кешує кожен об'єкт окремо під його власним URL.
- **Header Cleaning:** При створенні синтетичних відповідей видаляються заголовки `Content-Length`, щоб уникнути конфліктів розміру даних.

### Навігація та Стан
- **URL Sync:** Вибрані монстри та заклинання синхронізуються з URL (`?monster=slug`, `?spell=slug`).
- **History API:** Використовується `window.history.pushState` та слухач `popstate` для коректної роботи кнопок "Назад/Вперед".

### AI Промптінг
- **Типи запитів:** `campaign_plot` (сюжет), `scene_ideas` (сцени), `image` (описи для генераторів зображень).
- **Контекст:** AI отримує мінімізований JSON поточної сесії (опис, нотатки, існуючі сцени) для підтримки цілісності сюжету.

## 6. Правила розробки для ШІ
1. **Стилі:** Використовувати BEM. Змінні кольорів брати з `main.scss` (напр. `var(--accent)`).
2. **Лінтер:** Дотримуватися `eslint.config.js` (Flat Config).
3. **Локалізація:** Усі системні повідомлення та інтерфейс — українською мовою. Технічні дані D&D (назви монстрів/заклинань) — мовою оригіналу (English) з перекладом метаданих за можливості.
4. **Безпека:** API ключі лише через `.env`.
5. **Продуктивність:** Для довгих списків обов'язково використовувати `ReactList`.

## 7. Система іконок
Проєкт використовує єдиний компонент `Icon` для відображення графічних елементів.

### Компонент `Icon`
- **Пропси:**
  - `name` (string, обов'язковий): технічне ім'я іконки.
  - `size` (number): розмір іконки в пікселях (за замовчуванням зазвичай 24).
  - `strokeWidth`
  - `className` (string): додаткові класи для CSS (напр. `state-asc` для обертання).

Використання по типу <Icon name="wand" size={20} className="ComponentName__block-icon" />
